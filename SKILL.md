# Loop Engine — 流水线编排引擎

## 核心理念

把复杂流程拆成 **阶段（Phase）→ 门禁（Gate）→ ... → 汇总** 的标准化流水线。
执行模式由用户显式选择。循环（loop）和重试（local_retry）构成内外双层机制。
目标、约束、验收三要素驱动闭环。

**核心术语：**
- **Phase（阶段）**：流水线中的一个步骤。可以是调 skill、执行指令、合并数据、门禁校验、生成产物
- **Gate（门禁）**：质量校验点。启动独立 subagent 检查前一阶段的输出是否达标
- **Loop（外循环）**：质量不达标时回到前面的某个阶段，带着反馈改参数重新执行
- **Local Retry（内循环）**：当前阶段原地重试，输入不变，应对临时错误

---

## 设计取向（写给评估/使用这份 skill 的人，不是模型的执行指令）

这份 skill 是 prompt 模板，是设计独立调度系统的指南和引导——所有"必须""硬上限"最终都
靠执行它的模型自己兑现，没有外部程序计时计数。文档里凡是约束模型行为的规则，
都已经直接写在对应的字段说明或执行步骤里（用"你"做主语），而不是单独陈述成
一套哲学。这一节只是给人一个索引，说明几条贯穿全文的判断分别落在哪：

- gate 判断有方差、不可直接当 ground truth → 落在「gate 字段说明」的 `double_check`
- 循环终止时不该默认"最后一轮最好" → 落在「循环终止优先级」的 best-so-far 规则
- 重试没有反馈就是重复犯错 → 落在「循环 loop」的强制携带反馈要求
- 文档措辞要让模型对自己负责，而非误以为有外部机制兜底 → 落在「防执行幻觉纪律」

---

## 三种执行模式

用户在选择前需理解每种模式的执行规则：

### 串行 sequential

步骤依次执行，前一步完成才能启动下一步。

```
step-1 → [gate] → step-2 → [gate] → step-3 → generate
```

适用：步骤之间有明确前后依赖的场景。

### 并行 parallel

所有步骤同时启动，全部完成后再进入下一阶段。

```
step-A ──┐
step-B ──┤  同时启动，等待全部完成 → [gate] → generate
step-C ──┘
```

适用：多个独立任务互不依赖的场景。

### 混合 mixed

通过分组组合串行和并行。组内可并行或串行，组与组之间串行执行。

```
组1（并行）：step-A ──┐
             step-B ──┤  组1全部完成 → 组2（串行）：step-C → [gate] → generate
```

适用：第一步并行获取多源数据，第二步串行做依赖分析的场景。

## 工作流程

### 第一步：需求解析

要素分三层，**层与层的追问方式不同**：

**A. 骨架层（必须问，问错代价高，决定整条流水线的结构）**

骨架层内部分两轮问，不是三项一次性列给用户——执行模式是路由器，决定后两项
该怎么问（串行问"哪几步、顺序"；并行问"哪几个独立任务"；混合问"分几组、
组内并行还是串行"），所以必须先单独问、最先问：

**第一轮——只问执行模式：**

> "这条流水线打算怎么跑？" [ 串行 / 并行 / 混合 ]

**第二轮——在第一轮答案约束下问阶段列表 + 目标与验收**（范围已被收窄，
通常一轮问完）：

| 要素 | 说明 |
|:----|:-----|
| 阶段列表 | 有哪些步骤，每步用什么完成（skill / 自定义指令）——具体问法由执行模式决定，见上 |
| 目标与验收 | 什么算完成，怎么验证 |

**B. 调参层（先给默认值随草稿展示，不单独提问，用户在确认草稿时改）**

| 要素 | 默认值 |
|:----|:-----|
| local_retry（阶段内重试次数） | 2 |
| timeout_minutes | 30 |
| data_mode | overwrite（见下方"循环退出取值"，建议改 best_of） |
| on_failure | retry |
| 最终输出格式 | markdown |

设置 `max_iterations`、`timeout_minutes` 这类约束之前，先在草稿里给自己一个
判断（不需要单独问用户，按任务性质自行预判，写在 YAML 注释里供用户核对）：
**这条流水线是"错过本可以更好的结果代价更高"，还是"多耗资源继续跑代价更高"**。
前者倾向于把 `max_iterations` 设得宽松一些、`data_mode` 用 `best_of`；后者倾向于
设得紧、用 `overwrite` 更省事。这个判断没有客观正确答案，但必须显式做出来，
不能让默认值的数字看起来像是算出来的而实际只是抄了模板。

**C. 条件层（只有用户表述触发对应场景才问）**

| 要素 | 触发条件 |
|:----|:-----|
| 分组结构 | 用户选了混合模式才问 |
| 外循环 loop（from/back_to/trigger/break） | 用户提到"不达标重来""检查不过关再改"等信号才问 |
| 阶段间依赖 | 并行/混合且涉及 merge 时才问 |

**D. 开放层（两步问，不直接抛一个空白的开放问题）**

**第一步——轻量筛选，同一轮问两道并列的判断题**（环境支持按钮就用按钮，
不支持就用一句话是否）：

> 1. "这条流水线有什么你想让模型全程记住的方法/风格要求？（例如涉及数据
>    调研——来源要不要交叉验证；涉及内容生成——语气要不要统一，仅供参考，
>    按实际阶段类型换成相关的例子）" [ 有，我说说 / 没有，跳过 ]
>
> 2. "这条流水线运行有没有依赖你知道但模型不一定知道的外部条件？比如：
>    要在特定时间窗口内跑完、某个外部接口/网络访问是否受限、数据有没有
>    时效要求、外部系统是否有调用配额限制——这类信息只有你知道，模型
>    查不出来" [ 有，我说说 / 没有，跳过 ]

不熟悉该领域的典型坑就不要硬举例，直接问，不要用不相关的例子凑数。
两题负担都小，可以和 A 层收尾的问题放在同一轮，不违反节奏约束。两题
对应的代价不对等：第1题漏答，后续补一句 notes 就能修；第2题漏答，
往往要等流水线跑起来撞上才发现，代价高得多，因此即使用户对第1题选
"没有"，第2题也要照样单独问，不能因为第1题没东西就连带跳过。

**第二步——选"有"才展开：**

> "说说看，是什么？"

这一步完全开放、不限定范围，第1、2题各自分别展开（如果两题都选了
"有"，分两次问，不要合并成一个问题，避免用户的外部依赖回答被方法
偏好的回答顺序带偏）。第1题的回答收到后先检查是否包含 C 层触发信号
（如"不达标重来""检查不过关再改"）——是，按 C 层规则补问 loop 等
相关字段，这条内容进入 A 层骨架，不再当 notes 处理；不是，再逐条
套用「notes 字段说明」的两层判据分流：第一层判断是不是真额外（违反
了输出算不算错，算的话要落进 verification/check_list 而非 notes）；
通过第一层的，再按第二层判断进 pipeline 级 notes、phase 级 notes，
还是其实该直接写回某个 phase 的 instructions。不要把这条开放式回答
原文不加判断地整段塞进 notes。不要把这一步拆成具体子问题去问，开放
性是它的价值所在。

第2题的回答收到后，逐条标注到对应 phase 的 YAML 注释里（格式参考
「外部依赖自检」一节），不进 notes 家族——它是前提条件而非方法偏好，
两层判据不适用。

**两题都选"没有"——直接跳过，不追问，不强求。**

**节奏约束：**
- 每轮最多问 1-2 个问题，优先问 A 层第一轮（执行模式）。执行模式答完，
  紧接着问 A 层第二轮（阶段列表+目标验收）；A 层问完，紧接着问 D 层
  第一步两道筛选题（负担都小，可以和 A 层收尾问题同轮），均选"没有"
  则直接进入自检与生成草稿；任一题选"有"则先走完对应的 D 层第二步
  展开，再进入自检与生成草稿。B 层用默认值标注在草稿注释里，不要在
  生成草稿前再开一轮问卷
- 不要把 A/B/C/D 四层混在一条消息里一次性列给用户
- 如果用户提到"不断改进""优化到满意为止" → 主动建议补充目标和验收标准
- 如果用户用了"分析""调研""检查"等通用动词 → 不得默认为某个特定 skill，追问具体用什么来做
- 用户明确指定了 skill 名 → 直接使用；用户没指定 → 用 `custom` 类型
- A/D 两层问完（D 层视用户选择，可能只有两道筛选题一轮，也可能加一到
  两轮展开）→ 先做「结构风险自检」（见下），再生成 YAML 草稿，不再
  额外列一份要素清单等用户确认——YAML 草稿本身就是结构化后的确认
  载体，确认动作统一放在第三步，避免同一件事确认两次

> **💡 概念指引：** 表中标记 `→` 的字段在第二步中有详细定义和可选值说明（如「阶段类型」「循环 loop」「数据模式」「数据传递机制」「失败策略」），第一步只需理解基本含义，具体配置时查阅第二步对应章节。

#### 结构风险自检（生成草稿前，模型自查，不问用户）

A/D 两层问完、生成 YAML 草稿之前，对照下表自查一遍，命中项写成 YAML
注释挂在对应 phase 上，不命中不写注释（不要为了显得周全而堆砌无依据
的提醒）。这一步和 B 层同一性质——不单独占用户一轮，但必须做。

| 类别 | 自查什么 | 命中时注释怎么写 |
|:----|:----|:----|
| 跨阶段数据契约 | 下游 instructions 是否隐含依赖上游产出的具体格式/字段，而上游没有保证 | "⚠️ 本阶段期望输入为 X 格式，需确认上游 phase 实际产出符合" |
| 并行资源竞争 | 并行 phase 间是否写同一文件/调用同一限流接口 | "⚠️ 与 phase X 并行执行，两者均写入 Y，存在覆盖风险" |
| skill 隐性串行约束 | 放进 parallel 的 skill，是否实际上不能被并行多实例调用 | "⚠️ 该 skill 是否支持并行调用未经确认，建议核实" |
| gate 可验证性 | check_list 是否存在"违反了算错但看输出验证不出来"的项（如数据是否编造） | 按「notes 字段说明」特例，退回该 gate 的 notes，不进 check_list |
| merge 完整性 | `depends_on` 是否覆盖了所有需要的前置 phase | "⚠️ 请确认 depends_on 是否遗漏了某个前置 phase" |
| 外部依赖（可推断部分） | 见下方独立说明 | 见下方独立说明 |

**外部依赖自检，单独说明边界——这一项可靠性有限，不能当成把关结论：**

- 仅当某 phase 的 `type: skill` 指向一个实际存在的 skill 时才做：先
  view 该 skill 的 SKILL.md，根据文档内容判断是否涉及联网/外部接口/
  时效性数据，不能仅凭 skill 名称猜测
- 命中时注释必须指出依据（"根据该 skill 文档关于数据来源的描述，
  本阶段可能需要联网获取实时数据"），不能给不出具体依据的命中
- 收尾必须带非穷尽声明，不能呈现成干净的"已检查通过"：
  "（基于已读取的 skill 文档判断，非穷尽核查，网络权限/凭据等
  环境因素建议你自行确认）"
- 这一项只覆盖"模型能从文档里查到的"，查不到的外部依赖（时间窗口、
  外部系统的配额坑等）属于 D 层第一步第 2 题的范围，不是自检能补的——
  自检和那道追问题是互补关系，自检不能替代追问，追问也不能省略自检

### 第二步：定义流水线

#### 串行

```yaml
pipeline:
  name: "流水线名称"
  mode: sequential
  notes: |                # 跨阶段的额外要求（非核心成败标准），判据见
                           # 下方「notes 字段说明」，不是任意补充说明

  phases:
    - id: "step-1"
      name: "第一步"
      type: skill          # skill | custom | gate | merge | generate
      skill: "skill名"     # type=skill时填写具体 skill 名
      instructions: "做什么"
      notes: "本阶段专属的额外要求/背景说明（可省略），判据同上"
      local_retry: 2       # 内循环：失败时原地重试 2 次，输入不变

    - id: "check-1"
      name: "质量门禁"
      type: gate
      check: "gate-skill"  # 填写具体的门禁 skill 名
      check_list: "检查标准"
      partial_retry: false # 见「门禁（gate）说明」：是否支持只重做未通过的子项
      double_check: false  # 见「门禁（gate）说明」：高杠杆判断是否双重验证

    - id: "step-2"
      name: "第二步"
      type: custom
      instructions: "做什么"

    - id: "output"
      name: "输出"
      type: generate
      format: "markdown"   # html | markdown | json

  loops:
    - id: "loop-1"
      from: "check-1"             # 从哪个 gate 判断是否循环
      back_to: "step-1"           # 回到哪个阶段
      trigger: "gate 检查不通过"   # 触发条件
      break: "gate 通过 或 循环≥3次" # 跳出条件
      max_iterations: 3
      data_mode: "best_of"        # overwrite | accumulate | ask | best_of（推荐，见下）
      strategy: "根据门禁反馈调整参数"
      # 重新执行 back_to 指向的阶段时，必须显式带上：
      #   1) 上一轮 gate 给出的具体未通过项和证据
      #   2) 已经尝试过的调整方式（避免重复同一次失败的改法）
      # 这两项是强制行为，不是可选项，缺失会让循环退化为重复随机尝试

  goal:
    description: "完成标准"
    constraints:
      max_iterations: 3           # 全局循环上限
      timeout_minutes: 30         # 全局超时
      max_cost: null              # 可选：成本预算，单位为 subagent 调用次数（近似代理，
                                   # 非精确开销），并行+循环组合时尤其需要
    verification:
      method: "verification-skill" # 填写具体的验收 skill 名
      criteria: "验收标准"

  on_failure: retry                # retry | abort | skip | pause
  max_retries: 3                   # 重试预算总上限，与 phase 级 local_retry 不是同一个概念
```

#### 并行

```yaml
pipeline:
  name: "并行流水线"
  mode: parallel

  phases:
    - id: "task-a"
      name: "任务A"
      type: custom
      instructions: "做A"
      local_retry: 2

    - id: "task-b"
      name: "任务B"
      type: custom
      instructions: "做B"
      local_retry: 2

    - id: "merge-check"
      name: "归并验收"
      type: gate
      check: "gate-skill"
      check_list: "检查标准"
      wait_all: true               # 等所有前置阶段完成后再执行
      partial_retry: true          # 典型场景：check_list 能分别对应 task-a / task-b
                                    # 的输出时，未通过项只回退对应任务，不动已通过的那个

    - id: "output"
      name: "输出"
      type: generate
      format: "markdown"

  loops:
    - id: "loop-1"
      from: "merge-check"
      back_to: "task-a"            # 只回退 task-a；task-b 的已有输出保留不动
                                    # （此规则仅适用于 task-a/task-b 互不在同一并行组的情况；
                                    # 若 back_to 指向的 phase 属于某个并行组，组内全部重来，
                                    # 见「混合」schema 的跨组回退规则）
      trigger: "gate 检查不通过"
      break: "gate 通过 或 循环≥2次"
      max_iterations: 2
      data_mode: "best_of"
      strategy: "根据 check_list 中 task-a 对应项的反馈调整"

  on_failure: retry
  max_retries: 3
```

#### 混合

```yaml
pipeline:
  name: "混合流水线"
  mode: mixed

  groups:
    - id: "g1"
      parallel: true               # 组内并行
      phases:
        - id: "a"
          name: "任务A"
          type: custom
          instructions: "做A"
        - id: "b"
          name: "任务B"
          type: custom
          instructions: "做B"

    - id: "g2"
      parallel: false              # 组内串行
      phases:
        - id: "c"
          name: "综合处理"
          type: custom
          instructions: "做C"
        - id: "check-c"
          name: "质量门禁"
          type: gate
          check: "gate-skill"
          check_list: "综合处理的输出是否完整、准确，数据格式是否规范"
        - id: "output"
          name: "输出"
          type: generate
          format: "html"

  loops:
    - id: "refine"
      from: "check-c"              # 从 gate 节点判断是否触发循环
      back_to: "a"                 # 回到哪个 phase——注意："a"属于并行组g1
      trigger: "质量不通过"
      break: "质量通过 或 循环≥3次"
      max_iterations: 3
      data_mode: "best_of"
      # 跨组回退规则：back_to 指向的 phase 若属于某个并行组（如这里的 g1），
      # 循环会重新触发该组内的【全部】phase（a 和 b 一起重来），而不是只重跑
      # 被指名的那一个。原因：组内并行 phase 往往共享下游依赖（这里都流向 c），
      # 只重跑一个会让组内数据产生不一致的版本，比多花一次资源更危险

  on_failure: retry
  max_retries: 3
```

#### 阶段类型

| 类型 | 用途 | 行为 |
|:----|:-----|:------|
| `skill` | 调用另一个 skill | 启动 subagent 执行该 skill，传入指令和前置数据，等待返回 |
| `custom` | 执行自定义指令 | 直接执行给定的 prompt 或指令代码 |
| `gate` | 质量门禁 | 启动独立 subagent（不得由主流程自判），按 check_list 校验前置阶段输出 |
| `merge` | 合并多个输出 | 将多个前置阶段的输出合并为一个数据集，供下游使用 |
| `generate` | 生成最终产物 | 基于所有前置输出生成 html / markdown / json 格式的报告 |

**merge 示例：**

```yaml
# 在并行阶段后合并两个分析结果，用于汇总报告
- id: "merge-results"
  name: "合并分析结果"
  type: merge
  depends_on:                     # 明确指定依赖的上游 phase
    - "fraud-check"
    - "valuation"
  instructions: "将财务造假检测和估值分析的输出合并为一个结构化数据集"
```

#### notes 字段说明

判两层，顺序不能反：

**第一层——配不配进 notes 这个家族：**
违反这条信息，输出本身算不算错（不达标）？
- 算 → 不是"额外要求"，是核心要求，写进 `goal.verification.criteria`
  或 gate 的 `check_list`，被主动核查，不准放进 notes
- 不算（只是方法/过程上的偏好或限制，遵不遵守不直接决定输出对错）
  → 才是真正的额外要求，进入第二层
- 特例：理论上算"违反了输出就错"，但 gate 单看输出判断不出来
  （比如数据是否编造，看数字本身看不出来）→ 退回 notes，当作
  执行时必须遵守的约束，因为放进 check_list 也验不出来，没有意义

**第二层——通过第一层后，放哪：**

| 情况 | 进哪 |
|:----|:----|
| 管好几个 phase，要重复写 | pipeline 级 notes |
| 只管一个 phase，是背景/教训，不是任务本身 | 该 phase 的 notes |
| 单阶段 + 内容本身就是任务 | 直接进该 phase 的 instructions |

两层判据都和句子是正向还是负向无关，"要做X"和"不要做Y"一视同仁。
一句话里正反两面都说了，且反面就是正面的镜像（正面执行到位，
反面自动不会发生）→ 只留正面那半句，反面是废话，删掉。

**gate 字段说明：**

| 字段 | 必填 | 说明 |
|:----|:----|:------|
| `check` | 是 | 门禁校验使用的 skill 名，与普通 phase 的 `skill` 字段对应但语义不同（gate 执行校验而非执行任务） |
| `check_list` | 是 | 校验标准，可以是文本描述或条目列表 |
| `wait_all` | 否 | 是否等待所有前置 phase 完成后才执行（默认 false，并行场景建议设为 true） |
| `partial_retry` | 否 | check_list 多条且能归因到具体上游 phase 时（如 gate 跟在 merge 后面），是否只重做未通过项对应的 phase，而非整体回退（默认 false） |
| `double_check` | 否 | 该 gate 的判断是否高杠杆（直接决定终止/大幅返工）。为 true 时，独立跑两次 gate subagent，结果不一致则标记"无法验收"而非择一采信（默认 false）。只防偶发噪声，不防系统性偏差，见下方说明 |
| `notes` | 否 | 判据见「notes 字段说明」；对 gate 而言，能单独成为一条可勾选标准的必须写进 check_list；不能单独成立、只是依附某条 check_list 项的判断注解，或适用于整份 check_list 的总体判断原则，才进 notes（可省略） |

#### 内循环 local_retry vs 外循环 loop

local_retry 和 loop 本质是同一个模式（试 → 评 → 带着信息再试 → 设上限）在
不同粒度上的重复，不是两个互相独立的机制——区别只在于回退范围多大、反馈带得
多丰富。下面分开列是为了配置时字段清楚，理解时不必当成两套不同的东西。

| | local_retry（内循环） | loop（外循环） |
|:----|:----|:----|
| 范围 | 当前阶段原地 | 回到任意前置阶段 |
| 输入 | 不变 | 必须带着上一轮的具体失败点+已尝试调整（强制，见循环 loop 章节） |
| 适用 | 临时错误（网络波动、超时等） | 质量不达标需要改进参数或方法 |
| 优先级 | 先耗尽 local_retry，再触发 loop | loop 在 local_retry 失效后或 gate 不通过时触发 |
| 配置位置 | phase 级的 `local_retry: N` | pipeline 级的 `loops[]` |

**三个相似字段的区分（改名而非靠解释消歧）：**
- phase 级 `local_retry: 2` → 该阶段失败时原地重试的次数，输入不变
- 全局 `max_retries: 3`（建议读作 `retry_budget`）→ 整条流水线所有 local_retry 加总的上限，
  防止某个阶段反复失败拖垮全局预算
- 失败策略 `on_failure: retry` → 阶段失败且 local_retry/retry_budget 都耗尽后的默认处理方式
  （此处 retry 是策略名，与上面两个"次数"字段不是同一类东西）

优先级：先执行 phase 级 `local_retry`，耗尽后再检查全局 `max_retries`（retry_budget）。

#### 循环 loop

```yaml
loops:
  - id: "名称"
    from: "哪个 gate/phase"          # 从哪个节点判断是否要循环
    back_to: "回到哪个 phase"        # 回到哪个阶段重新执行
    trigger: "触发条件"              # 什么条件下进入循环
    break: "跳出条件"               # 什么条件下退出循环
    max_iterations: 3               # 最大循环次数（loop 级）
    data_mode: "best_of"            # overwrite | accumulate | ask | best_of
    strategy: "每次怎么调整参数"      # 描述性指引，由模型具体执行
```

**强制要求（不是可选项）：** 每次因 loop 触发重新执行 `back_to` 指向的阶段时，
必须把以下两项作为该阶段执行时的显式输入，而不是只把 `strategy` 当成一句口号：
1. 上一轮 gate 给出的具体未通过项和证据（不是"不通过"三个字，是哪一条、为什么）
2. 此前各轮已经尝试过的调整方式（防止新一轮无意识地重复同一个无效改法）

没有这两项，重新执行在认知上和上一轮没有区别，循环只是在消耗预算，不是在收敛。

#### 循环终止优先级

循环可能因多种原因终止，按优先级从高到低：

```
1. 目标达成（goal 的 verification 通过）→ 正常完成，优先级最高
2. loop.max_iterations 耗尽         → 强制终止，标记"循环次数耗尽"
3. goal.constraints.timeout 超时     → 强制终止，标记"全局超时"
4. goal.constraints.max_cost 耗尽    → 强制终止，标记"成本预算耗尽"（如设置了该约束）
5. 用户手动打断                     → 强制终止，标记"用户中断"
```

优先级 1 是正常终止；2-5 是异常终止，均需在汇总报告中说明原因。
注意区分两个 max_iterations：loop 级的控制单次循环上限，goal.constraints 级的控制全局上限。
全局上限优先于 loop 级上限。

**收敛判断（次数之外的第二条准绳）：** 次数/时间/成本只回答"还能不能再来一轮"，
不回答"这几轮是不是在变好"。每完成一轮，都要把这一轮 gate 的结果和此前各轮做对比：
- 如果本轮比此前最好的一轮更优 → 更新"best-so-far"指针为本轮
- 如果本轮比 best-so-far 差 → 不更新指针，但仍计入 max_iterations 消耗
循环因 2-5 强制终止时，最终交付 **best-so-far 指向的那一轮**，而不是无条件交付
最后一轮的结果——除非最后一轮恰好就是 best-so-far。这一点必须在汇总报告中注明
"最终交付的是第几轮的结果，以及为什么不是最后一轮"。

#### 数据模式

| 模式 | 行为 | 首次执行 | 适用场景 |
|:----|:-----|:--------|:---------|
| `overwrite` | 循环时覆盖前一轮输出，下游只看最新 | 正常输出 | 最终只需要最优结果（且确信越往后越好） |
| `accumulate` | 保留各轮输出，按迭代号归档 | 正常输出 | 需要对比各轮改进效果 |
| `ask` | 询问用户如何处理 | 正常输出 | 不确定场景，交给用户决定 |
| `best_of` | 保留各轮输出，循环结束时交付 best-so-far 指向的那一轮（见「循环终止优先级」） | 正常输出 | 默认推荐：不假设"最后一轮一定最好" |

**首次执行**指循环触发前的第一次执行，四种模式行为相同——正常产生输出。
循环触发后，overwrite 覆盖、accumulate 追加、ask 询问用户、best_of 按 best-so-far 指针决定最终交付物（但和 accumulate 一样保留各轮记录，便于回溯）。

#### 数据传递机制

phase 之间的输出传递遵循以下规则。**注意：这里没有独立的"执行引擎"，
以下动作都由你（当前执行此流水线的模型）亲自用 bash/文件工具完成：**

1. 每个 phase 完成后，你需要把输出写入一个临时文件，并在你维护的执行记录里
   写下这个文件路径——这份"审计日志"就是你自己持续追加的一份记录文件，不是
   外部系统自动生成的
2. 下游 phase 通过 `depends_on` 或分组结构确定上游，你需要主动去读取上游的
   输出文件再继续
3. 数据量明显小（几百字以内）时可以直接在上下文里传内容；数据量大时，传文件
   路径，由下游阶段自己读取——不必纠结精确字节数，判断标准是"塞进上下文是否
   合理"，而不是某个具体阈值
4. `merge` 类型时，你需要主动把所有前置 phase 的输出汇总进一个数据集，这一步
   没有自动化，要靠你逐个读取后手动合并
5. 循环时按 data_mode 处理：overwrite 由你覆盖旧文件，accumulate 由你按迭代号
   归档为不同文件名，best_of 由你额外维护一个"当前最优指向哪个文件"的记录

#### 失败策略

| 策略 | 行为 | 默认值 |
|:----|:-----|:-------|
| `retry` | 重试，最多 max_retries 次 | 默认 |
| `abort` | 中止整个流水线，保留已完成的输出 | — |
| `skip` | 跳过当前阶段，标记为"已跳过"，继续执行 | — |
| `pause` | 标记为"需人工介入"，暂停流水线，等待用户指令 | — |

未声明 `on_failure` 时，默认值为 `retry`。
注意：`skip` 策略可能导致下游数据缺失，使用前需确认下游是否有容错处理。

#### 门禁（gate）说明

- gate 是独立的 quality phase，由 `type: gate` 标识
- gate **必须**启动独立 subagent 执行校验，不得由主流程模型自判
- 校验标准由 `check_list` 字段定义（文本描述或条目列表）；`notes` 只装两类内容：
  ①依附某条具体 check_list 项、脱离那条单独看没有意义的判断注解（如"来源写
  '网络'不算标注"，这是在解释"是否标注来源"这条该怎么判，不是独立标准）；
  ②适用于整份 check_list、不专属任何一条的总体判断原则（如"模糊不清一律
  判不通过"）。能脱离 check_list 单独成为一条可勾选标准的，必须明写进
  check_list，不能靠 notes 隐性传递
- 校验结果必须写入迭代日志，包含：通过/不通过、评分（如有）、详细证据
- gate 自身执行失败（如 subagent 崩溃）→ 按 `on_failure` 策略处理
- 如果门禁校验标准不明确（如只说"检查质量"没说具体指标），标记为"无法验收"而非"已通过"
- **gate 的判断本身有方差，不是 ground truth。** 一次 subagent 调用给出的"通过/不通过"
  是一次性的模型判断，可能因为措辞、上下文长度、随机性而和重跑一次的结果不一致。
  对于决定是否触发大幅返工或终止流水线的高杠杆 gate，把 `double_check` 设为 true：
  独立跑两次校验，两次结论一致才采信；不一致时标记"无法验收"，不要择一采信，
  也不要自己居中调和——分歧本身就是信号，应该交给用户或更高权限的判断
- **`double_check` 只防噪声，不防偏差。** 噪声是"这次判断飘了"（偶发、随机），
  偏差是"这个 check 标准本身就系统性偏松/偏严/有盲区"（持续、结构性）。同一个
  check skill 独立跑两次，抓的是前者——两次一致只能说明这次没手滑，不能说明
  标准本身没问题，反而容易被误读成"更可信"。怀疑的是偏差时，应该换一个不同的
  check skill 或不同的校验方法做交叉验证，而不是把同一个 check skill 再跑一次
- 设置 `partial_retry: true` 且 check_list 多条标准能各自归因到具体上游 phase 时
  （典型场景：gate 跟在 merge 之后，check_list 第N条对应第N个被合并的 phase），
  循环回退只重做未通过项对应的 phase，其余已通过的 phase 不动；如果归因不清楚
  （check_list 是笼统的整体描述），仍按整 phase 回退处理，不要强行拆分

### ⚙️ 防执行幻觉纪律（横跨全流程）

下面每一条的主语都是**你**——当前正在执行这条流水线的模型——而不是某个独立
存在的"执行引擎"。没有人在背后替你计时计数，全靠你自己老实记录和自我约束。

1. **你来做策略判断，但"做了没有"要靠你留下可核查的痕迹。** 你决定"要不要
   触发循环""回到哪个阶段""怎么调参数"，但 `max_iterations`、`timeout` 这类
   上限不是摆设——你必须在每次进入循环前，显式数清楚已经循环了几次、用掉了多少
   时间，写在你的执行记录里再决定能不能继续。如果你判断"再来一轮肯定能成"，
   这个判断本身恰恰是要被次数上限挡住的东西，不能作为突破上限的理由

2. **门禁判断不能自己对自己说"过了"。** gate 必须以独立 subagent 的身份执行
   校验，不能是主流程的你顺手自己判一句"达标了"。如果你发现自己在同一个推理
   链条里既执行了任务又给任务打了通过分，这就是要避免的情况

3. **目标达成要靠你能拿出来的证据，不能靠你自己说"做到了"。** 验收必须落在
   文件、数据、评分记录这些可以被回头检查的东西上。如果你没办法拿出这类证据
   证明目标达成，老实标记"无法验收"，不要因为流程快走完了就倾向于宣称"已达标"

4. **你的执行轨迹要让别人能看懂你做了什么。** 每个 phase 的输出你要写入临时
   文件，每次 gate 的结果你要写进校验记录，每次循环时给下游的反馈你要写进日志。
   最终的汇总报告要能让用户单看这份报告，就重建出你这一路是怎么走的，而不需要
   反过来相信你的一句话总结

### 第三步：配置确认与后续决策

流水线配置完成后，**不要直接进入执行阶段**。先走完确认和决策流程：

#### 一、展示配置 + 用户确认

将完整的流水线配置（YAML）展示给用户，用清晰的流程图说明整体结构（建议用 ASCII 图或 Mermaid）：

```
例：
① 下载财报（web-access）
       │
       ▼
├── ② 财务造假检测 ├── ③ 费米估值 ├── ④ 管理层分析  ← 并行
       │
       ▼
⑤ 对抗验证（大门将）
       │
       ├── 发现问题 → 循环重回②/③/④（1次）
       │
       ▼
⑥ 生成 HTML 报告
```

**确认清单**（逐项与用户核对）：
- [ ] 执行模式（串行/并行/混合）是否正确
- [ ] 阶段列表和顺序是否完整
- [ ] 各阶段的 type 和 skill 名是否正确
- [ ] 循环条件（trigger / break / max_iterations）是否合意
- [ ] 数据模式（overwrite / accumulate / ask / best_of）是否合意
- [ ] 目标、约束、验收标准是否清晰
- [ ] 失败策略是否符合预期
- [ ] B 层调参字段（local_retry / timeout / 输出格式）的默认值是否需要调整
- [ ] D 层第 1 题（方法偏好）补充说明是否被正确分流——触发 C 层的（如提到
      不达标重来）没有被错判成 notes 而漏掉 loop 定义；该进
      verification/check_list 的没有被错放进 notes；该进 notes 的没有被
      错放进某个 phase 的 instructions
- [ ] 高杠杆 gate 是否需要打开 `double_check`；多上游 gate 是否需要打开 `partial_retry`
- [ ] D 层第 2 题（外部依赖）如果用户选了"有"，每一条是否都已落到对应
      phase 的注释里，没有被遗漏或误判成 notes
- [ ] 「结构风险自检」标出的注释逐条过一遍——尤其外部依赖那条，自检
      明确声明非穷尽，网络权限/凭据等环境因素仍需用户自己确认

如果用户提出修改，更新 YAML 后再次展示确认，直到用户说"可以"。

#### 二、决策：立即执行 vs 持久化为 Skill

配置确认后，**必须**问用户：

> **流水线已设计完成。你是想：**
> 
> **A) 🚀 立即执行** — 现在就跑这条流水线
> 
> **B) 📦 持久化为 Skill** — 保存为可复用的 skill，以后随时调用

**选择 A（立即执行）**：
- 告知用户即将开始执行，提及预计耗时（如有经验值）
- 进入第四步：执行

**选择 B（持久化为 Skill）**：
- 告知用户将启动 skill-creator 来生成 SKILL.md
- 在传给 skill-creator 时附上以下信息：
  - 完整的 YAML 配置（或结构化的阶段描述——如果流水线涉及多个 skill 编排，结构化描述通常比纯 YAML 更易读、更易扩展）
  - 执行模式、分组结构、循环逻辑
  - 各阶段的 type、skill 名、指令
  - 目标、约束、验收标准
  - 数据传递规则和失败策略
- skill-creator 完成后，**再次询问用户是否要执行这个流水线**（因为现在有了 skill，可以更方便地重复跑）
- 如果用户说"要跑"，进入第四步

#### 三、设计阶段的特殊处理

如果用户在需求解析阶段（第一步）就表示"帮我设计一个流水线并保存为 skill"，可以：
- 跳过追问"立即执行 vs 持久化"的环节（因为用户已经表态）
- 直接说明：好的，那我来完成设计后直接用 skill-creator 打包
- 但仍需完成配置展示和确认（一、展示配置 + 用户确认）

### 第四步：执行

下面这套逻辑由**你**（执行此流水线的模型）亲自走一遍，没有外部"执行引擎"替你
跑这些步骤——你既是调度者也是执行者，这一点贯穿全部 10 步：

```
1. 解析流水线配置，校验必填字段
   - 校验字段完整性（name、mode、phases 或 groups 必须存在）
   - 校验 mode 值是否为 sequential / parallel / mixed
   - 校验 phases 中的 type 值是否为 skill / custom / gate / merge / generate
   - 校验失败 → 向用户展示缺失/错误的字段，请求修正后重试，不默认使用任何值
2. 根据 mode 确定执行策略：
   - sequential → 按 phases[] 顺序逐个执行
   - parallel → 所有 phases[] 同时启动
   - mixed → 按 groups[] 顺序逐一处理每组，组内按各组 parallel 字段指定的策略
3. 每个 phase 启动 subagent 之前：
   - 把 pipeline 级 notes（全文）与本 phase 的 notes 一并拼入该 subagent
     的输入，作为 instructions 之外的额外约束；gate 类型同理：
     check_list + pipeline 级 notes + 本 phase notes 一并传给执行
     校验的 subagent
   - 随后 run_in_background → 等待完成
   - 如果 subagent 依赖的 skill 不存在（如 `type: skill` 但 skill 名不匹配）→ 降级为
     `type: custom`，用 instructions 中的引导自行完成该阶段，并明确告知用户发生了
     降级（这是唯一允许的"自行决定"，且必须主动声明，区别于第1步的字段缺失——
     字段缺失是结构性错误必须拒绝，skill 找不到是可恢复的执行细节可以降级，
     但两者都不能悄悄发生）
   - 如果 subagent 依赖的 skill 存在但执行失败 → 走第4步的失败处理流程
4. 处理 phase 自身失败（如 subagent 崩溃、超时、返回错误）：
   - 原地重试前先检查：这次失败之前，该 phase 是否已经造成了不会自动撤销的
     副作用（写了一半的文件、调用过的外部接口、发出去的通知等）。有的话先
     处理掉（清理半成品文件 / 标记已调用过避免重复调用 / 不重发已发出的通知），
     不能默认重试一定是从干净状态开始
   - 检查该 phase 的 local_retry 次数 → 未耗尽则原地重试，输入不变
   - 耗尽后检查全局 max_retries（retry_budget）→ 未耗尽则继续重试
   - 全部耗尽 → 按 on_failure 策略处理
5. phase 成功后 → 写入输出文件 → 你自己在执行记录里追加一条记录（即「审计日志」）
6. 如有 gate → 启动独立 subagent（不是你顺手自判）做校验
   - 若该 gate 的 `double_check: true` → 独立跑两次，结果不一致则标记"无法验收"，
     不要自己择一采信。注意 double_check 只能抓出"判断本身飘了"这种偶发噪声，
     抓不出"这个 check 标准本身就系统性偏松/偏严"这类系统性偏差——同一个
     check skill 跑两次一致，只能说明这次没手滑，不能说明标准本身没问题。
     怀疑的是后者时，应换一个不同的 check skill 或校验方法，而不是重复同一个
7. gate 校验完成：
   - 校验通过 → 记录本轮为候选结果，与当前 best-so-far 比较，更优则更新指针
     → 继续到下一阶段
   - 校验不通过 → 先判断这次不通过的性质：
     - **可通过调整恢复**（参数不合适、指令不够具体等）→ 同样记录本轮结果用于
       和 best-so-far 比较 → 查是否有 loop 定义，按下方循环逻辑处理
     - **结构性、不可恢复**（依赖的数据源根本不存在、上游 phase 缺了必需输入、
       任务本身的前提条件不成立等——换参数、换instructions都解决不了）→
       不触发 loop，无论 loop 是否定义都直接按 on_failure 策略处理，并在报告
       里说明判定为"结构性失败"的依据。循环存在的前提是"再试一次有可能更好"，
       这个前提不成立时硬塞进 loop 只是在重复消耗预算
     - 不确定属于哪一种 → 按可恢复处理，进入 loop，但在循环反馈里明确写出
       "本轮无法判断是否结构性"，留给后续轮次或用户判断
   - 进入 loop 的情况，检查 break 条件（目标达成、次数耗尽、超时、成本耗尽）
     - 满足 break → 退出循环，交付 best-so-far 指向的那一轮，继续
     - 不满足 break → 检查 max_iterations 是否耗尽
       - 未耗尽 → 若 `partial_retry: true` 且能归因到具体 phase，只回退重做
         对应 phase；否则回到 back_to 指向的完整阶段（若 back_to 落在并行组内，
         整组一起重来，见「混合」schema 的跨组回退规则）。重新执行时必须携带
         上一轮的未通过项证据和已尝试过的调整（见「循环 loop」强制要求）
       - 已耗尽 → 强制退出循环，标记"循环上限耗尽"，交付 best-so-far
8. gate 自身执行失败（进程崩溃、超时等）→ 按 on_failure 策略处理
9. 所有 phase 完成 → 执行 generate 类型，基于 best-so-far 指向的数据生成最终产物
10. 输出汇总报告
```

**约束校验时机：**
- 进入循环前：检查 `loop.max_iterations` 是否耗尽
- 每次循环开始前：检查 `goal.constraints.max_iterations` 是否耗尽，以及
  `goal.constraints.max_cost`（如设置）是否耗尽
- 持续监控：`goal.constraints.timeout_minutes` 是否超时——这件事没有人替你计时，
  你需要在每个关键节点（每个 phase 完成、每次循环开始）主动记一下经过的时间，
  自己判断是否超时
- 持续计数：`goal.constraints.max_cost`（如设置）同样没有人替你计数，每启动
  一次 subagent（含 phase 执行和 gate 校验）记 1 次调用，在同样的关键节点
  累加，自己判断是否耗尽。这是近似代理而非精确开销，汇总报告里如实标注为
  "调用次数"，不要包装成"成本"或具体金额

**用户中断处理：**
- 如果用户中途打断流水线（如输入新指令或要求停止）：
  1. 立即中止当前正在运行的 phase
  2. 收集已完成的 phase 输出（不丢弃）
  3. 生成部分汇总报告，含已完成/未完成/进行中的阶段状态
  4. 在报告中说明中断原因（标记为"用户中断"）
  5. 告知用户已保存的部分结果位置，并询问是否需要后续继续

### 第五步：生成汇总报告

```markdown
# 执行报告 — [流水线名称]

## 总体
- 模式：串行/并行/混合
- 阶段：N | 成功：N | 失败：N | 跳过：N
- 循环：N 次 | 总耗时：Xs
- 目标达成：✅/❌
- 最终交付：第 N 轮结果（best-so-far）| 是否等于最后一轮：是/否（不等于时说明原因）

## 各阶段
| 阶段 | 轮次 | 状态 | 耗时 | 输出 |
|:----|:----|:----|:----|:----|

## 执行轨迹
- 审计日志：xxx
- 门禁记录：xxx（含 double_check 是否触发及两次结果是否一致）
- 循环反馈：xxx（每轮携带的未通过项+已尝试调整摘要）

## 约束遵守情况
- max_iterations（loop 级）：已用/上限
- max_iterations（全局级）：已用/上限
- timeout：耗时/上限
- max_cost（如设置，单位为调用次数，近似代理）：已用/上限
- 是否违反约束：是/否

## 补充说明遵守情况
- pipeline 级 notes：逐条列出每一条在哪个 phase 被实际应用过，
  给不出具体应用位置的标记"无法核实"，不要笼统打勾"遵守"
- 各 phase 级 notes：同上
```

> 如果循环存在收敛趋势异常（如来回震荡而非逐轮改善），在「总体」下额外加一行
> 说明，便于用户判断是 gate 标准本身有问题还是任务确实难收敛。

---

### 第六步：报告交付与反馈收集

汇总报告生成后，**必须**做以下三件事：

1. **展示给用户** — 告知用户报告已生成，说明文件保存路径和报告结构
2. **询问反馈** — "这份报告是否符合你的预期？是否有需要调整或补充的地方？"
3. **引导下一步** — 根据用户反馈决定：
   - 用户满意 → 结束流水线
   - 用户不满 → 询问具体需要改进什么，判断是否需要重新执行某阶段或调整配置
   - 用户想优化描述触发词 → 建议使用 skill-creator 的 description optimizer

> **注意：** 如果流水线执行过程中发生了循环重做、阶段失败或异常终止，在展示报告时**必须主动说明**这些情况，不可只展示"好看的部分"。

## 已知局限

1. 这是 prompt 模板，不是真实存在的独立调度系统——所有"必须""硬上限"最终都
   靠执行它的模型自觉兑现，文档语言已尽量改为对模型自身的元认知要求，但终究
   没有外部机制能强制拦截模型的偷懒或自我宽松
2. 大数据量传路径而非内容，但"大"没有精确阈值，需要模型自行判断是否塞进上下文合理
3. 不修改被调 skill 的行为，skill 本身的问题会影响流水线
4. gate 依赖明确校验标准，否则可能流于形式；`double_check` 只能降低噪声（偶发
   判断波动），不能纠正偏差（check 标准本身系统性偏松/偏严）——同一个 check
   skill 的系统性弱点，跑两次大概率还是错在同一个地方，这种情况下两次"一致"
   反而会被误读成更可信
5. 并行模式下不硬编码并发上限，由系统负载自动控制
6. `best_of` 依赖 gate 给出的评分/通过情况做轮次间比较，如果 gate 本身只输出
   二元通过/不通过、不输出可比较的程度信息，"哪一轮更优"在未通过的几轮之间
   可能无法区分，此时退化为"最近一次未通过的结果"
7. `partial_retry` 依赖 check_list 能清晰归因到具体上游 phase，多数单 phase 后
   直接跟一个 gate 的场景里，partial_retry 等价于没有效果（因为只有一个 phase
   可回退），它真正发挥作用的场景是 merge 后的多源 gate
8. 「结构风险自检」中外部依赖一项只能覆盖"被调 skill 文档里写明的"外部条件，
   查不到的（时间窗口、外部系统的配额坑、下游消费方的格式要求等只存在用户
   经验里的信息）必须靠 D 层第一步第 2 题主动问出来，自检不能替代这道题；
   反过来，用户也可能漏答这道题——两个机制都不是穷尽保证，只是把遗漏概率
   降到比"完全不查不问"更低
